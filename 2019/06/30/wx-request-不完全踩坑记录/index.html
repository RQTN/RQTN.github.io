前言

在一个网站里面发起 https 请求是很常见的，但是，由于微信小程序是腾讯内部的产品，不能直接打开一个外部的链接。所以，对于 客户端与服务端接口的对接 ，我们只有使用微信小程序提供的 wx.request() 的 API 接口实现 前端后台的对接。

但在实际的使用过程中，仍然能不小心踩到在官方文档中并未强调或提及的这样那样的坑，在此对使用 wx.request 可能遇到的问题进行总结和记录。

关于 data 参数说明

在官方文档中已有记录

最终发送给服务器的数据是 String 类型，如果传入的 data 不是 String 类型，会被转换成 String 。转换规则如下：

  • 对于 GET 方法的数据，会将数据转换成 query string（encodeURIComponent(k)=encodeURIComponent(v)&encodeURIComponent(k)=encodeURIComponent(v)...）
  • 对于 POST 方法且 header['content-type'] 为 application/json 的数据，会对数据进行 JSON 序列化
  • 对于 POST 方法且 header['content-type'] 为 application/x-www-form-urlencoded 的数据，会将数据转换成 query string （encodeURIComponent(k)=encodeURIComponent(v)&encodeURIComponent(k)=encodeURIComponent(v)...）

关于 method

微信小程序许多地方都 严格区分大小写，比如 wx.request() 的 method 的 value 就必须为大写。默认的 method 值为 GET，如果需要使用其他 method 值，需要显式地在 wx.request 中提供。method 的所有合法值如下表所示：

值 说明
OPTIONS 返回服务器针对特定资源所支持的 HTTP 请求方法。
GET 向特定的资源发出请求。 一般为查询请求。
HEAD 向服务器索要与 GET 请求相一致的响应，只不过响应体将不会被返回。
POST 向指定资源提交数据进行处理请求。数据被包含在请求体中。一般为修改请求。
PUT 向指定资源位置上传其最新内容。 一般为新增请求。
DELETE 请求服务器删除 Request-URI 所标识的资源。一般为删除请求。
TRACE 回显服务器收到的请求，主要用于测试或诊断，或可用于服务器网络测试。
CONNECT HTTP/1.1 协议中预留给能够将连接改为管道方式的代理服务器。

关于接口调用 http 和 https 的问题

正式发布上线的小程序必须设置 https 地址，但在本地开发中，微信公共平台支持使用 http 测试项目：将项目设置中的 tls 安全检测关闭，便可以使用 127.0.0.1 或者其他地址测试。

为保证数据传输安全，提高业务安全性，使用 https 方式调用。

关于微信小程序网络请求数限制

对于比较复杂的应用，往往一个功能的实现需要请求发送多个请求，而一个微信小程序，同时只能有 10 个网络请求连接。解决方案：

管理一个请求队列，使同时请求数始终不大于 10。不直接使用 wx.request 请求网络资源。已经有人造好这个轮子了：https://github.com/zhengjunxin/wx-promise-request

关于 wx.request 的异步

微信提供的 wx.request 方法并没有提供异步处理，但是微信小程序是支持 ES6 语法的，故可以使用 Promise 稍加改造，以避免回调地狱，你可以自己用 Promise 包装 wx.request，不过同样是上面的链接，已经有人造好轮子了：https://github.com/zhengjunxin/wx-promise-request

关于 Referer

小程序规定 header 中不能设置 Referer，格式固定为：

https://servicewechat.com/{appid}/{version}/page-frame.html

其中 {appid} 为小程序的 appid，{version} 为小程序的版本号，版本号为 0 表示为开发版。利用这个设置，可以轻松识别 api 接口来源。例如，你是一家小程序第三方开发服务器，给众多小程序提供了接口，依靠 referer 可以轻松识别请求来源。
