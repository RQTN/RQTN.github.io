简答题

  • 用简短的语言给出对分析、设计的理解。

    参考课件 LarmanOOAD_Part1_C01.pdf P8-P9

    • Analysis

      • emphasizes an investigation of the problem and requirements, rather than a solution. For example, if a new online trading system is desired, how will it be used? What are its functions?

        强调对问题和需求的调查，而不是解决方案。例如，如果需要（开发）一个新的在线交易系统，该系统将被如何使用呢？它的功能应该是什么呢？

      • do the right thing

        分析让我们做对（最应该做）的事情

    • Design

      • emphasizes a conceptual solution (in software and hardware) that fulfills the requirements, rather than its implementation. For example, a description of a database schema and software objects.

        强调一个概念性的解决方案（软件上的和硬件上的），方案能够满足需求，而不是其实现。例如，数据库架构和软件对象的描述。

      • do the thing right

        设计让我们把（最应该做的）事情做对

  • 用一句话描述面向对象的分析与设计的优势。

    参考课件 LarmanOOAD_Part1_C01.pdf P13-P14

    • the analyst don’t have to be “language experts”

      分析师不一定得是“语言专家”

      • the experts in the problem domain and the implementation-level experts can communicate using a common notation, e.g. the same kinds of modeling notations.

        问题域的专家（分析师）和实现级别的专家可以使用通用的符号来进行交流通信，例如，相同类型的建模符号。

  • 简述 UML（统一建模语言）的作用。考试考哪些图？

    参考课件 LarmanOOAD_Part1_C01.pdf P21-P27

    • The UML is standard diagramming language to visualize the results of analysis and design.

      UML 是标准的图表语言，用于可视化分析和设计的结果。

    • Three Ways to Apply UML

      • UML as sketch
      • UML as blueprint
      • UML as programming language
      • Agile modeling

    • Three Perspectives to Apply UML

      • Conceptual perspective
      • Specification (software) perspective
      • Implementation (software) perspective

    • UML Overview
      感觉会考：

      • 用例图
      • 静态图
        • 类图

  • 从软件本质的角度，解释软件范围（需求）控制的可行性

    • 软件的本质：

      • complexity 复杂性
      • conformity 一致性
      • changeability 可变性
      • invisibility 不可视性

    • 解释软件范围（需求）控制的可行性

      • Software requirements are prone to frequent change
      • In many cases, requirements allocated to software must be renegotiated and reprioritized
        • Small changes to software can have large negative effects（There are no small software changes）
        • The term prototyping has different connotations for systems engineers and software engineers
      • Software often provides the interfaces that interconnect other system components
      • Cyber security is a present and growing concern for systems that incorporate software
      • Several Pareto 80-20 distributions apply to software
        • 20% 功能能满足 80% 人的需要
        • 20% 测试可发现 80% 错误

      前 3 条指出在不一致、不可视、易于变化的环境下定义软件定义（需求）风险。
      接着 3 条是需要面对的各种设计风险，不仅要能满足软件的变化，而且软件的非功能性需求的实现成本远远高于功能实现成本。
      最后一条是做工程做要理解的之一，软件分析不是发现更多的功能与需求，而是确认有效的功能与需求，去伪存真不断满足客户需要。软件设计也不是追求最先进技术，而是控制变化附加的成本，使得软件生产的预算、时间等能在掌控之中。

      综上可知，软件本身的复杂性，一致性，可变性和不可视性决定了软件开发的困难，而且软件的范围（需求）实际上在不一致，不可视，易于变化的环境实际上是非常多变的，但这并不意味着我们就不去做软件范围（需求）控制，反而是要建立高效和相对完善的机制，确保软件范围（需求）的分析能够及时更新，不断调整软件发展的方向（比如采用迭代的方法）。总而言之，软件范围（需求）控制势在必行
